# TASK-001 开发贪吃蛇小游戏

## 目标
在当前仓库中实现一个可直接在浏览器打开运行的贪吃蛇小游戏，零依赖、无构建步骤、单机可玩。游戏需要包含随机障碍物机制：地图在每局开始时生成若干静态障碍物，蛇撞到障碍物即判定失败。整体实现要边界清晰、代码结构简单、便于后续作为 DevFlow 示例任务复用。

## 范围与假设
- 采用纯 HTML / CSS / JavaScript 实现，不引入框架、打包器或第三方依赖。
- 以桌面浏览器为主，支持常见键盘操作；移动端不作为本任务主验收条件。
- 游戏核心能力包括：开始、暂停、重开、计分、碰撞结束、食物刷新、速度递增、随机障碍物。
- 页面应能通过直接打开 `index.html` 运行，不依赖本地服务。
- 障碍物在一局游戏内保持静态，不随游戏进程移动或重生成，避免规则歧义。
- 视觉风格以清晰可用为主，避免过度设计，保持演示性质和可维护性。

## 方案选择
- 渲染方式：使用 `canvas` 实现游戏区域，便于控制帧更新、碰撞检测和绘制性能。
- 状态管理：使用显式状态对象管理游戏状态，避免隐式全局变量。
- 输入方式：键盘方向键和 `WASD`，并支持空格暂停/继续、`R` 重新开始。
- 障碍物策略：每局开始或重开时一次性生成静态障碍物，数量由网格大小决定并设上限，确保可玩性。
- 占用管理：维护统一的占用集合或判定函数，生成食物和障碍物时都通过同一套空位判断，避免重叠。
- 碰撞规则：蛇头进入障碍物格子即失败，和撞墙、自撞同等处理为结束态。
- 代码组织：按“初始化、输入、更新、渲染、结束态处理”拆分逻辑，保证职责单一。

## 文件布局
```text
/
├── index.html
├── styles.css
├── game.js
└── README.md
```

- `index.html`：承载页面结构、游戏画布、HUD、操作说明。
- `styles.css`：页面布局、响应式样式、游戏面板样式。
- `game.js`：游戏主逻辑、渲染循环、输入处理、状态机、障碍物与食物生成。
- `README.md`：运行方式、控制说明、游戏规则和障碍物规则说明。

## 实现切片

### Slice 1: 页面骨架与运行入口
- 搭建基础 HTML 结构，包含标题、分数区域、状态提示、画布容器、操作说明。
- 引入 `styles.css` 与 `game.js`，保证文件可直接双击打开运行。
- 确认页面初始状态清晰，能在无交互时展示“开始提示”。

### Slice 2: 游戏状态与初始化
- 定义游戏状态对象，包括网格尺寸、蛇身数组、当前方向、下一方向、食物位置、障碍物列表、分数、速度、是否运行、是否结束。
- 实现初始化函数，负责生成初始蛇、初始食物、初始障碍物和默认状态。
- 统一处理“开始 / 暂停 / 重开”状态切换，避免逻辑分散。
- 明确障碍物生成所需的基础参数，例如网格大小、障碍物数量上限和最小可用空位阈值。

### Slice 3: 随机障碍物与占用约束
- 在每局初始化时生成静态障碍物，数量按网格规模计算并进行上限控制，保证剩余可玩空间充足。
- 生成逻辑必须避开蛇初始位置和初始食物位置，不能产生重叠。
- 食物生成逻辑必须避开蛇身和所有障碍物，必要时使用“随机采样 + 失败回退扫描”的方式保证一定能找到空位。
- 若地图空间过小导致无法放置预期数量的障碍物，应自动降低障碍物数量而不是报错或卡死。
- 将障碍物纳入统一占用判断，避免后续逻辑重复实现。

### Slice 4: 核心游戏循环
- 实现固定步长更新逻辑，按时间推进蛇移动。
- 处理方向变更，禁止立即反向导致自撞的非法输入。
- 实现吃到食物后增长蛇身、加分、刷新食物、提高速度。
- 实现碰撞检测，包括撞墙、撞到自己、撞到障碍物。
- 明确碰撞顺序：本次移动生成新蛇头后，先判断越界，再判断自撞和障碍物命中，再处理食物命中。

### Slice 5: 渲染与 UI 反馈
- 在 `canvas` 中渲染网格、蛇、食物和障碍物。
- 障碍物采用与蛇和食物清晰区分的视觉样式，确保失败原因可一眼识别。
- 同步更新分数、最高提示、暂停状态和结束状态文案。
- 在游戏结束后显示明确的重开提示，说明撞墙、自撞或撞障碍物都属于失败。

### Slice 6: 交互完善与边界处理
- 支持键盘控制：方向键、`WASD`、空格、`R`。
- 处理窗口尺寸变化对画布显示的影响，保持居中与适配。
- 避免重复按键造成状态错乱，确保输入在每个 tick 内只生效一次。
- 对非法状态转换做保护，例如未开始时按暂停、结束态按方向键等。
- 保证重开时会重新生成整套状态，包括蛇、食物、障碍物和速度初值。

### Slice 7: 文档与示例说明
- 在 `README.md` 中说明如何打开运行、控制方式、游戏规则和障碍物规则。
- 说明障碍物是静态生成且不会和蛇身、食物重叠，方便后续作为 DevFlow 示例参考。
- 如仓库已有 DevFlow 约定，则在不改变现有约定的前提下补充任务说明链接或引用。

## 验证方式
- 手动验证 `index.html` 直接打开可运行，无控制台报错。
- 验证方向控制正常，且不允许瞬间反向。
- 验证吃食物后蛇身增长、分数增加、食物重新生成且不会刷在蛇身上或障碍物上。
- 验证随机障碍物在每局开始和重开时生成，且不会出现在蛇初始位置或初始食物位置。
- 验证撞墙、撞自己和撞障碍物都会进入结束态。
- 验证结束态文案能明确提示失败原因与重开操作。
- 验证空格可暂停/继续，`R` 可重开。
- 验证不同窗口宽度下布局不溢出，画布和说明区域显示正常。
- 补充一个最小静态检查步骤，确认 `game.js` 无语法错误。

## 风险与开放决策
- 障碍物数量过多会显著压缩可玩空间，因此必须设置基于网格规模的上限和最小留白策略。
- 食物和障碍物同时需要避让，占用判断若实现不统一，容易出现重叠或死循环，所以应集中在同一套空位选择逻辑里。
- 随机采样在后期可能变慢，尤其是可用空位减少时，因此需要在采样失败后回退到顺序扫描，保证稳定性。
- 碰撞判定的顺序必须固定，否则蛇头在同一格同时命中食物和障碍物时会出现规则歧义；本计划规定障碍物命中优先触发失败。
- 若后续希望加入关卡化或动态障碍物，可在当前静态障碍物方案上扩展，但本任务不引入该复杂度。

## 完成标准
- 仓库中存在一个无需构建即可运行的贪吃蛇小游戏。
- 用户可以通过浏览器直接打开并完成完整一局游戏。
- 随机障碍物已实现，且蛇撞到障碍物会直接失败。
- 食物、蛇身和障碍物之间不存在非法重叠。
- 代码结构清晰，主要逻辑集中在少量文件中，便于后续迭代和作为示例任务复用。
